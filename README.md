<h1>REST API сервис -Серверная часть для мобильного приложения iTalk (.Net/MVC5)</h1>


Что вкратце представляет собой сервис:
Это сервис взаимодействия клиентов и консультантов. 
Консультанты могут оказывать консультации, клиенты подписываться на них.
Определяется время консультации и клиентом, и консультантом
Клиент и консультант может общаться по аудио/видео связи и с помощью чата
Регистрация/авторизация организуется через популярные соцсети и сервисы
Авторизация так же должна быть реализована на стороне сервера встроенными средставми ASP.Net

Мобильное приложение iTalk
Программное обеспечение по настоящему техническому заданию, это:

1. Мобильное приложение “iTalk” для платформы Android в двух версиях - версии для Клиентов и версии для Консультантов со
взаимодополняющим функционалом. Пользователь может выбрать, кем он хочет выступать в приложении: клиентом или
консультантом. В зависимости от выбора пользователь может скачать либо версию для клиента, либо версию для
консультантов.
Приложения с одной стороны позволяют лицам (консультантам), чья услуга может быть оказана посредством аудио или видео
связи, предлагать свои услуги неограниченному кругу лиц, а с другой стороны лицам (клиентам), заинтересованным в
получении услуги, которая может быть оказана посредством аудио или видео связи, получать доступ к такой услуге. В
приложении должен быть реализован функционал текстового мини-чата между Клиентом и Консультантом, а также
функционал приема платежей Клиентов и выгрузки информации для оплаты услуг Консультантов.

2. Сервер с базой данных, который позволяет хранить данные, необходимые для работы приложения и удаленно
взаимодействовать с ними (в рамках ТЗ).

3. Административная панель (реализованная как веб-приложение), которая позволяет управлять работой программного
обеспечения, содержимым базы данных и просматривать статистику (по настоящему ТЗ).

4. Промо-лэндинг приложения для привлечения потенциальных пользователей и информационных сообщений о работе
сервиса.

13 ОФОРМЛЕНИЕ ЗАКАЗА
На данном экране пользователь производит оформление заказа путем
заполнения формы заказа. Форма заказа представляет собой следующие
элементы интерфейса:
- Выбор услуги из списка
- Комментарии к заказу
- Время, удобное для созвона
- Выбор аудио или видео консультация
- Кнопка “Оформить заказ”
После нажатия на кнопку “Оформить заказ” информация отправляется
консультанту, который в свою очередь должен отреагировать на заявку в течение
3 минут (если в течение этого времени отклик не производится, заявка
аннулируется). После нажатия также открывается текстовый чат, где можно
согласовать какие-то вопросы. Отклик консультанта представляет собой
отдельный экран на котором выводится информация о времени проведения
консультации, которое предложил пользователь, либо предложение другого
времени проведения консультации. Также пользователь может получить отказ в
проведении консультации от консультанта.
После проведения консультации пользователю предлагается поставить оценку
консультанту, а также добавить свой комментарий.
Используемые графические элементы не являются конечным вариантом дизайна.

14 ОФОРМЛЕНИЕ ЗАКАЗА
После нажатия на кнопку “Оформить заказ” информация отправляется

Кнопки доступа к основным функциям приложения должны быть спроектированы с учетом антропометрической
совместимости: пространственная компоновка форм и размерные характеристики кнопок должны обеспечивать быстрый
доступ к функциям приложения пальцами рук без использования стилуса.

49 ПОДДЕРЖКА ЯЗЫКОВ
Мобильное приложение iTalk
Приложение №1 к Договору № 497 от 29 июня 2018 года
Приложение должно быть событийно-ориентированным: не должно быть циклов ожидания изменения состояния, ожидание
наступления событий и активизация обработчиков должна быть организована с помощью системных средств.
Система должна стабильно работать при одновременном осуществлении не менее 50 транзакций (заказы, установление
соединений, видео/аудио связь, платежи и т.д.)

50 ТРЕБОВАНИЯ К ПРОИЗВОДИТЕЛЬНОСТИ
Мобильное приложение iTalk
Приложение №1 к Договору № 497 от 29 июня 2018 года
Приложение не должно допускать непредусмотренных завершений вследствие активности системных служб среды

Приложение не должно допускать непредусмотренных завершений вследствие реакции на системные события, связанные с

Приложение должно периодически сохранять данные с которыми работает пользователь для возможности их восстановления
после непредусмотренных завершений.

51 ТРЕБОВАНИЯ К НАДЕЖНОСТИ

52 ТРЕБОВАНИЯ К ДОСТУПНОСТИ
Мобильное приложение iTalk
Приложение №1 к Договору № 497 от 29 июня 2018 года
Связь серверной и клиентской стороны должна осуществляться по HTTPS запросам. Все запросы должны быть защищены от
просмотра третьей стороной с применение TLS протокола. Организация клиент-серверной связки должна исключать
возможность осуществления всех видов технических атак на получение доступа к серверу и на получение информации при
передаче между клиентской и серверной сторонами.
Серверная сторона должна учитывать версию мобильного приложения и изменять логику своей работы в том случае, если это
необходимо для поддержки нескольких версий мобильного приложения.
Взаимодействие серверной и клиентской сторон должно быть организовано таким образом, чтобы незначительные
функциональные изменения серверной стороны не приводили к необходимости обновления мобильного приложения.
Обмен запросами между клиентской и серверной стороной организуется по REST-стандарту, что подразумевает собой
соответствующую структуру наименования запросов и логику использования POST, GET, PUT, DELETE и PATCH методов.
